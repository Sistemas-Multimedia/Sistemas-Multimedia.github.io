1 Description

1.1 Intra (I) video coding

In the III... (or Intra video) coding, the 2D block-DWT, the 2D DWT, or any other spatial transform, is used on sequences of frames (images) to exploit the spatial correlation. This is achieved by simply iterating the spatial decorrelation as it is described in the Algorithm 1 [1], where V in the input sequence and S controls the number of SRLs (Spatial Resolution Levels)1 . The synthesis transform is computed using the Algorithm 2. In the Fig. 1 there is an example of the decomposition generated for three frames V 0, V 1 and V 2.

Algorithm 1: III-coding(V /* original video sequence */, S /* Number of extra levels */) → (O /* transformed video sequence */)

  1. O = {} /* empty sequence */.
  2. for Vi ∈V:
    1. Oi ←2D-TS(Vi) /* 2D analysis spatial transform */.

Algorithm 2: III-decoding(O /* transformed video sequence */, S /* Number of extra levels */) → (V /* original video sequence */)

  1. V = {} /* empty sequence */.
  2. for Oi ∈O:
    1. Vi ←2D-T−S(Oi) /* 2D synthesis spatial transform */.

1.2 Motion compensation prediction (P) video coding

III... video coding is fast and good for video edition, but the compression ratios are poor because the temporal correlation is not removed. IPP... video coding (also called inter-coding) uses MC (Motion Compensation) to exploit the temporal correlation in frame sequences. IPP... encoders split the sequence into GOPs (Groups of Frames), where the ﬁrst frame of each GOP is a I-type frame, and the rest of frames of the GOP are P-type. P-frames are motion-compensated frames and usually the reference frame is the previous one, in the temporal order.

1.3 Types of macro-blocks

A P-type frame is created in two steps:

  1. The frame is divided into non-overlapping blocks (typically of 16 × 16 pixels), that in this context we will call, macro-blocks.
  2. A prediction frame is generated using a ﬁeld of motion vectors (one vector per matro-block) and a reference.
  3. The prediction frame is substracted to the predicted frame, generating a prediction-error frame.

In general, most of the time this procedure success in temporally decorrelate the frames. However, it can also happen that when the reference(s) frame(s) is very diﬀerent from the predicted one, the prediction-error frame has a higher entropy than the original (predicted) one.

In order to minimize this inconvenient, most video coding standards use the following types of macro-blocks:

  1. I (intra): if the macro-block is not compensated.
  2. P (predicted): if the macro-blocks is compensated.
  3. B (bidirectionally predicted): if the macro-block is compensated using an anterior and a posterior frame.
  4. S (skipped): if the macro-block is substituted by reference one (no substraction is performed).

The decision of the type of each macro-block is performed at the compressor, and this information is signaled in the code-stream. An visual example of the decision of the type of the macro-blocks is shown in the Figure 2.

1.4 Macro-block type decision

Ideally, the type (also called, mode) of the macro-blocks is decided considering all the posibilities (I, P, B, or S) and selecting the most beneﬁcial one from a RD perpective, i.e., selecting the alternative that minimizes the RD cost. This RDO is performed at the macro-block level.

W = color-T(Vk), (a)

M(k − 1) → k = ME(Wk−1,Wk), (c)

where the symbol − represents to the pixel-wise substraction,

Ek ∼ = QE(Ek), (g)

where QE(⋅) represents the lossy compression of the prediction error texture data,

E ∼k = DQ−1(E ∼ k), (h)

where DQ−1(⋅) represents the decompression of the prediction error texture data,

W ∼k ←E ∼k + W ∧k, (i)

and notice that if W ∧k = 0, then E ∼k = W ∼k,

W ∧k = P(W ∼k−1,M(k − 1) → k), (j)

where P(⋅,⋅) is a motion compensated predictor, and

W ∧k−1 = Z−1(W ∧,k − 1), (k)

where by deﬁnition, Z−1(W ∧,−1) = 0, and

V = color-T−1(W k), (l)

is the inverse color transform.

Notice that if W ∧k is similar to Wk, then Ek will be approximately zero, and therefore, easely compressed. Another interesting aspect to highlight is that the encoder replicates de decoder in order to use the reconstructed images as reference and avoid the drift error.
